build queue coordinator in /app/scheduler/ is broken. the simulate(config, events) function runs but doesnt handle most of the logic correctly

config dict has max_concurrent (int), default_timeout_ms (int), retry_limit (int), priority_order (list of priority strings highest to lowest), resource_pools (dict pool name to capacity), priority_boost_ms (int for pending build aging), priority_decay_ms (int for running build decay), preemption_enabled (bool)

events are dicts with ts (timestamp int) and kind string. submit events have build_id, priority (from priority_order), optional dependencies list of build_ids, optional timeout_ms, optional resources dict of pool to count needed, optional preemptible bool defaulting true. start_next tries starting highest effective priority eligible pending build, preempting lowest priority running preemptible build if needed when preemption_enabled and pending build has strictly higher effective priority. complete and fail mark builds done releasing resources, fail retries if under retry_limit going back to pending with retry_count incremented and submit_ts preserved. cancel cancels build plus all builds depending on it transitively. timeout_check fails running builds where ts >= start_ts + timeout_ms

build states are pending running completed failed cancelled. eligible means pending with all deps completed and resources available. effective priority for pending uses priority_boost_ms aging from submit, for running uses priority_decay_ms from start, both capped at priority_order bounds. same effective priority uses submission order for pending, start order for running. preempted builds go back to pending preserving submit_ts and retry_count without incrementing. builds with failed or cancelled deps get cancelled. duplicate build_ids ignored, unknown build_ids ignored, complete/fail on non-running ignored

bugs are spread across modules: dependencies.py doesnt check deps or track dependents. coordinator.py doesnt do preemption or retry logic or cancel propagation
